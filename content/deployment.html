Belajar Deployment

Pelajari cara membawa aplikasi dari localhost ke production: hosting, CI/CD, Docker, cloud platforms, dan best practices.

1

Apa itu Deployment?

Deployment adalah proses memindahkan aplikasi dari environment development ke environment production agar bisa diakses oleh user.

DEVELOPMENT               STAGING                  PRODUCTION
─────────────             ────────                 ──────────
┌───────────┐             ┌───────────┐            ┌───────────┐
│  Localhost│             │  Test     │            │  Live     │
│  laptop   │────────────►│  Server   │───────────►│  Server   │
│           │             │           │            │           │
│  .env.dev │             │  .env.stg │            │  .env.prod│
└───────────┘             └───────────┘            └───────────┘
    │                          │                        │
    ▼                          ▼                        ▼
 localhost:3000           staging.app.com          app.com
Environment dalam Development:
1. Development — Di komputer lokal, untuk coding dan testing awal
2. Staging — Server testing, mirip production untuk final testing
3. Production — Server live yang diakses user
2

Deployment Options: Static vs Server

Cara deployment berbeda tergantung jenis aplikasi. Static sites lebih mudah dan murah, sedangkan server-side apps membutuhkan lebih banyak resource.

Static Sites

  • HTML, CSS, JS files only
  • No server-side processing
  • Cheap or free hosting
  • CDN delivery
  • Vercel, Netlify, GitHub Pages

Server-Side Apps

  • Node.js, Python, PHP, etc
  • Need runtime environment
  • Higher cost
  • More configuration
  • Heroku, Railway, DigitalOcean, AWS
3

Deploy ke Vercel

Vercel adalah platform deployment terbaik untuk frontend frameworks (Next.js, React, Vue, dll) dengan fitur auto-deploy dari Git.

Terminal
# Install Vercel CLI
npm install -g vercel

# Login
vercel login

# Deploy dari local
vercel

# Deploy to production
vercel --prod

# Untuk Next.js, build otomatis!
# Vercel detect framework secara otomatis
vercel.json (optional config)
{
  "version": 2,
  "builds": [
    { "src": "package.json", "use": "@vercel/next" }
  ],
  "routes": [
    { "src": "/(.*)", "dest": "/" }
  ],
  "env": {
    "NEXT_PUBLIC_API_URL": "https://api.example.com"
  }
}
Kelebihan Vercel:
- Free tier generous
- Auto-deploy dari Git push
- Preview deployment untuk setiap branch
- Built-in SSL/HTTPS
- Global CDN
- Perfect untuk Next.js
4

Deploy ke Netlify

Netlify adalah alternatif Vercel dengan fitur serupa, bagus untuk static sites dan JAMstack applications.

Terminal
# Install Netlify CLI
npm install -g netlify-cli

# Login
netlify login

# Initialize project
netlify init

# Deploy preview
netlify deploy

# Deploy to production
netlify deploy --prod
netlify.toml
[build]
  command = "npm run build"
  publish = "dist"

[build.environment]
  NODE_VERSION = "18"

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200

[[headers]]
  for = "/*"
  [headers.values]
    X-Frame-Options = "DENY"
    X-Content-Type-Options = "nosniff"

Deploy ke GitHub Pages

GitHub Pages adalah hosting gratis dari GitHub untuk static sites. Cocok untuk portfolio, documentation, dan project pages.

Terminal
# Method 1: Manual - push ke branch gh-pages
git checkout -b gh-pages
git push origin gh-pages

# Method 2: GitHub Actions (recommended)
# .github/workflows/deploy.yml

name: Deploy to GitHub Pages

on:
  push:
    branches: [ main ]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '18'
      - run: npm ci
      - run: npm run build
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./dist
GitHub Pages URL:
- User site: username.github.io
- Project site: username.github.io/repo-name

Backend Deployment: Railway & Render

Untuk aplikasi backend (Node.js, Python, dll), Railway dan Render adalah PaaS (Platform as a Service) yang mudah digunakan.

Terminal - Railway
# Install Railway CLI
npm install -g @railway/cli

# Login
railway login

# Initialize project
railway init

# Deploy
railway up

# Add database
railway add --plugin postgres

# Set environment variables
railway variables set JWT_SECRET=your-secret

# Open deployed app
railway open
render.yaml
services:
  - type: web
    name: my-api
    env: node
    buildCommand: npm install
    startCommand: npm start
    envVars:
      - key: NODE_ENV
        value: production
      - key: DATABASE_URL
        fromDatabase:
          name: my-db
          property: connectionString

databases:
  - name: my-db
    plan: free

Environment Variables di Production

Environment variables berbeda di setiap environment. Jangan hardcode sensitive values!

Config Pattern
// config.js - Centralized config
const config = {
  development: {
    apiUrl: 'http://localhost:3000/api',
    dbUrl: 'mongodb://localhost:27017/dev'
  },
  production: {
    apiUrl: process.env.API_URL,
    dbUrl: process.env.DATABASE_URL
  }
};

const env = process.env.NODE_ENV || 'development';
export default config[env];

// Di hosting platform, set via dashboard:
// Vercel: Settings → Environment Variables
// Railway: Variables tab
// Render: Environment section

// Validasi env vars saat startup
const requiredEnvVars = [
  'DATABASE_URL',
  'JWT_SECRET',
  'API_KEY'
];

requiredEnvVars.forEach(envVar => {
  if (!process.env[envVar]) {
    throw new Error(`Missing required env var: ${envVar}`);
  }
});
Best Practices:
- Jangan commit .env ke Git
- Gunakan different values untuk setiap environment
- Rotate secrets secara berkala
- Gunakan secrets manager untuk production besar

Build Process & Optimization

Build process mengubah source code menjadi file yang optimal untuk production.

package.json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview",
    "lint": "eslint src --ext .js,.jsx",
    "test": "vitest"
  }
}

# Typical deployment commands
npm ci            # Clean install (faster, uses lock file)
npm run build     # Generate production build
npm run preview   # Test build locally
vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  build: {
    outDir: 'dist',
    sourcemap: false,  // Disable for production
    minify: 'terser',
    rollupOptions: {
      output: {
        manualChunks: {
          'vendor': ['react', 'react-dom'],
          'utils': ['lodash', 'axios']
        }
      }
    }
  }
});

CI/CD Pipeline

CI/CD (Continuous Integration/Continuous Deployment) mengotomatiskan proses testing dan deployment setiap ada perubahan kode.

.github/workflows/ci.yml
name: CI/CD Pipeline

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Run linter
        run: npm run lint
      
      - name: Run tests
        run: npm test
      
      - name: Build
        run: npm run build

  deploy:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v3
      
      - name: Deploy to Vercel
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.ORG_ID }}
          vercel-project-id: ${{ secrets.PROJECT_ID }}
          vercel-args: '--prod'

Docker: Containerization

Docker memungkinkan package aplikasi dengan semua dependencies dalam container yang konsisten di semua environment.

Dockerfile
# Node.js application Dockerfile
FROM node:18-alpine

# Set working directory
WORKDIR /app

# Copy package files
COPY package*.json ./

# Install dependencies
RUN npm ci --only=production

# Copy source code
COPY . .

# Expose port
EXPOSE 3000

# Start command
CMD ["node", "server.js"]

# Multi-stage build (smaller image)
FROM node:18-alpine AS builder
WORKDIR /app
COPY . .
RUN npm install && npm run build

FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
CMD ["node", "dist/server.js"]
Terminal
# Build image
docker build -t myapp:latest .

# Run container
docker run -p 3000:3000 --env-file .env myapp:latest

# Run in background
docker run -d -p 3000:3000 --name myapp myapp:latest

# View logs
docker logs myapp -f

# Stop & remove
docker stop myapp
docker rm myapp

Docker Compose: Multi-Container

Docker Compose memudahkan menjalankan aplikasi multi-container (app + database + redis, dll).

docker-compose.yml
version: '3.8'

services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/myapp
      REDIS_URL: redis://redis:6379
    depends_on:
      - db
      - redis

  db:
    image: postgres:15-alpine
    volumes:
      - postgres_data:/var/lib/postgresql/data
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: myapp

  redis:
    image: redis:7-alpine
    volumes:
      - redis_data:/data

volumes:
  postgres_data:
  redis_data:
Terminal
# Start all services
docker-compose up -d

# View logs
docker-compose logs -f

# Stop all
docker-compose down

# Stop & remove volumes
docker-compose down -v

# Rebuild after code changes
docker-compose up --build

Custom Domain & DNS

Custom domain membuat aplikasi terlihat profesional. DNS menghubungkan domain dengan server.

DNS Records
# Common DNS Record Types

A Record      # Domain → IP Address
example.com     →  192.168.1.1

CNAME         # Domain → Another Domain
www.example.com →  example.com
app.example.com →  myapp.vercel.app

TXT           # Text records (verification, SPF)
example.com     →  "v=spf1 include:_spf.google.com ~all"

MX             # Mail server
example.com     →  mail.example.com

# Setup custom domain di Vercel/Netlify:
# 1. Add domain di dashboard
# 2. Copy DNS records
# 3. Add records di domain registrar
# 4. Wait for propagation (minutes to 48 hours)
Domain Registrar Populer:
- Namecheap (recommended)
- Google Domains
- Cloudflare (free, dengan proxy)
- GoDaddy

SSL/HTTPS Configuration

HTTPS adalah keharusan untuk production. SSL certificate mengenkripsi data antara browser dan server.

SSL di Platform Modern:
- Vercel/Netlify — Auto SSL via Let's Encrypt, tinggal add domain
- Cloudflare — Free SSL dengan proxy
- Certbot — Manual SSL untuk VPS
Terminal (Certbot di VPS)
# Install Certbot
sudo apt update
sudo apt install certbot python3-certbot-nginx

# Get certificate
sudo certbot --nginx -d example.com -d www.example.com

# Auto-renewal test
sudo certbot renew --dry-run

# Nginx config dengan SSL
server {
    listen 443 ssl;
    server_name example.com;
    
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;
}

server {
    listen 80;
    server_name example.com;
    return 301 https://$server_name$request_uri;
}

Monitoring & Logging

Monitoring membantu mengetahui kesehatan aplikasi, sedangkan logging penting untuk debugging.

Monitoring Tools:
- Sentry — Error tracking & performance
- Datadog — Full observability
- New Relic — APM & monitoring
- Uptime Robot — Free uptime monitoring
- Grafana + Prometheus — Self-hosted monitoring
JavaScript (Sentry)
// Install: npm install @sentry/node
const Sentry = require('@sentry/node');

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  environment: process.env.NODE_ENV,
  tracesSampleRate: 0.1
});

// Error tracking
try {
  // Code that might fail
} catch (error) {
  Sentry.captureException(error);
  throw error;
}

// Performance monitoring
const transaction = Sentry.startTransaction({
  op: "db.query",
  name: "Get Users"
});
// ... query code
transaction.finish();

Performance Optimization

Optimasi performa membuat aplikasi lebih cepat dan menghemat resource.

Minify & compress assets (JS, CSS, images)
Enable gzip/brotli compression
Use CDN for static files
Implement caching headers
Lazy load images
Code splitting for JS bundles
Database query optimization
Connection pooling
Nginx Config
# Enable gzip
gzip on;
gzip_types text/plain text/css application/json application/javascript text/xml;
gzip_min_length 1000;

# Cache static files
location ~* \.(jpg|jpeg|png|gif|ico|css|js)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
}

# Security headers
add_header X-Frame-Options "SAMEORIGIN";
add_header X-Content-Type-Options "nosniff";
add_header X-XSS-Protection "1; mode=block";

Production Security Checklist

HTTPS enabled (SSL certificate)
Environment variables for secrets
Rate limiting implemented
CORS configured properly
Input validation on all endpoints
SQL injection prevention
XSS protection headers
Authentication & authorization
Regular dependency updates
Backup strategy in place
Express Security Middleware
const helmet = require('helmet');
const rateLimit = require('express-rate-limit');

// Security headers
app.use(helmet());

// Rate limiting
app.use(rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100
}));

// CORS
app.use(cors({
  origin: ['https://yourdomain.com'],
  credentials: true
}));

// Disable x-powered-by header
app.disable('x-powered-by');

Memilih Platform Deployment

Vercel

  • Best for: Next.js, React
  • Free tier generous
  • Auto-deploy from Git
  • Edge functions

Netlify

  • Best for: Static sites
  • Free tier available
  • Forms, Functions
  • Great for JAMstack

Railway

  • Best for: Full-stack apps
  • $5 free credits/month
  • Database included
  • Easy Docker deploy

DigitalOcean

  • Best for: VPS control
  • Starts $4/month
  • Full server access
  • Docker/K8s support

AWS

  • Best for: Enterprise
  • Pay per use
  • Most features
  • Steeper learning curve

Cloudflare Pages

  • Best for: Static + Workers
  • Free tier generous
  • Global CDN
  • Workers for backend
