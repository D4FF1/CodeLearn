Belajar Git & GitHub

Kuasai version control: dari command dasar hingga collaboration workflow. Setiap materi dilengkapi penjelasan mendalam dan visualisasi.

1

Apa itu Version Control & Git?

Version Control System (VCS) adalah sistem yang mencatat setiap perubahan pada file sepanjang waktu. Git adalah VCS terdistribusi yang paling populer, diciptakan oleh Linus Torvalds pada 2005 untuk pengembangan kernel Linux.

Kenapa Git Penting?
1. History — Bisa kembali ke versi manapun kapan saja
2. Collaboration — Banyak orang bisa kerja di project yang sama tanpa saling menimpa
3. Branching — Bisa eksperimen fitur baru tanpa rusak kode utama
4. Backup — Setiap clone adalah backup penuh dari repository
5. Accountability — Tahu siapa mengubah apa dan kapan 
TANPA GIT                          DENGAN GIT
─────────                          ──────────
file_v1.txt                        commit abc123 (HEAD)
file_v2.txt                    →   commit def456
file_final.txt                     commit ghi789
file_final_v2.txt                  ...history lengkap
file_final_fix.txt
Git vs GitHub:
Git = Software version control yang diinstall di komputer (local)
GitHub = Platform hosting repository Git di cloud (remote). Alternatif: GitLab, Bitbucket, Gitea.
2

Instalasi & Konfigurasi Awal

Setelah menginstall Git, langkah pertama adalah mengkonfigurasi identitas. Ini penting karena setiap commit akan mencatat siapa yang membuat perubahan.

Penjelasan: Git menyimpan konfigurasi di tiga level:
1. --system → semua user di komputer ini
2. --global → user saat ini (paling sering dipakai)
3. --local → repository tertentu saja
Gunakan --global untuk setup yang berlaku di semua project.
Terminal
# Cek versi Git
git --version

# Set nama dan email (WAJIB)
git config --global user.name "Nama Kamu"
git config --global user.email "email@example.com"

# Lihat semua konfigurasi
git config --list

# Set editor default (opsional)
git config --global core.editor "code --wait"

# Set branch default ke main
git config --global init.defaultBranch main
Tip: Pastikan email yang didaftarkan sama dengan email GitHub/GitLab agar commit terhubung dengan akun kamu.
3

Membuat Repository: git init

Repository (repo) adalah folder project yang dilacak oleh Git. Perintah git init membuat folder menjadi repository Git.

Apa yang terjadi saat git init?
Git membuat folder tersembunyi .git yang berisi semua data version control. Folder ini jangan dihapus atau di-edit manual! Tanpa .git, folder biasa tidak akan dilacak.
Terminal
# Buat folder project baru
mkdir my-project
cd my-project

# Inisialisasi repository
git init
# Output: Initialized empty Git repository in /path/my-project/.git/

# Cek status
git status
# Output: On branch main, No commits yet, nothing to commit
SEBELUM git init          SESUDAH git init
────────────────          ────────────────
my-project/               my-project/
├── index.html            ├── .git/          ← folder git (tersembunyi)
├── style.css             ├── index.html
└── script.js             ├── style.css
                           └── script.js
4

Memantau Perubahan: git status

git status adalah perintah paling sering digunakan untuk melihat kondisi repository: file apa yang berubah, apa yang sudah di-stage, dan posisi branch.

3 State File di Git:
1. Untracked — File baru, belum pernah dilacak Git (merah)
2. Modified — File sudah dilacak tapi ada perubahan (merah)
3. Staged — File siap di-commit (hijau)
4. Unmodified — File sama dengan di repo (tidak muncul di status)
Terminal
git status

# Output contoh:
On branch main
No commits yet

Untracked files:
  index.html    # merah = belum di-track
  style.css

nothing added to commit but untracked files present
Tip: Gunakan git status -s untuk output lebih singkat (short format).
5

Staging File: git add

git add memindahkan file dari "working directory" ke "staging area". Hanya file yang di-stage yang akan di-commit. Ini memungkinkan kamu memilih perubahan mana yang ingin disimpan.

Konsep 3 Area Git:
Working Directory → tempat kamu edit file
Staging Area (Index) → antrian file yang akan di-commit
Repository (.git) → database permanen semua snapshot

git add = Working → Staging
git commit = Staging → Repository
Terminal
# Tambah satu file
git add index.html

# Tambah beberapa file
git add index.html style.css script.js

# Tambah semua file (yang sudah berubah)
git add .

# Tambah semua file di seluruh folder
git add -A

# Tambah dengan interaktif (pilih bagian kode)
git add -p filename.js

# Lihat apa yang sudah di-stage
git status
git diff --staged
         git add                  git commit
Working ──────────► Staging ──────────────► Repository
Directory           Area                    (.git)

index.html ──► [staged: index.html] ──► [commit: "add index"]
6

Menyimpan Snapshot: git commit

Commit adalah "checkpoint" atau snapshot dari project pada waktu tertentu. Setiap commit punya ID unik (hash SHA-1), penulis, waktu, dan pesan yang menjelaskan perubahan.

Aturan Pesan Commit yang Baik:
1. Gunakan bahasa Inggris atau Indonesia, tapi konsisten
2. Awali dengan kata kerja: "Add", "Fix", "Update", "Remove", "Refactor"
3. Baris pertama max 50 karakter, jelas dan spesifik
4. Jangan tulis "update", "fix bug" — terlalu umum!
5. Contoh baik: "Fix login validation on mobile devices"
Terminal
# Commit dengan pesan
git commit -m "Add homepage layout"

# Commit dengan editor (untuk pesan panjang)
git commit

# Commit dengan pesan multi-baris
git commit -m "Add user authentication" -m "Includes login, logout, and session management"

# Skip staging (hanya file yang sudah di-track)
git commit -a -m "Update styles"

# Ubah commit terakhir (SEBELUM push!)
git commit --amend -m "New message"
Peringatan: Jangan ubah commit yang sudah di-push ke remote! Akan menyebabkan konflik dengan tim.
7

Melihat History: git log

git log menampilkan riwayat semua commit. Setiap commit ditampilkan dengan hash, author, date, dan message.

Terminal
# Log standar
git log

# Log ringkas (satu baris per commit)
git log --oneline

# Log dengan grafik branch
git log --oneline --graph --all

# Log dengan perubahan file
git log --stat

# Log dengan diff lengkap
git log -p

# Log 5 commit terakhir
git log -5

# Log berdasarkan author
git log --author="John"

# Log berdasarkan tanggal
git log --since="2024-01-01" --until="2024-12-31"

# Log dengan format custom
git log --pretty=format:"%h - %an : %s"
git log --oneline --graph --all

* abc1234 (HEAD -> main) Add footer component
* def5678 Update navbar styles
* ghi9012 Add responsive layout
|\
| * jkl3456 (feature/auth) Add login form
| * mno7890 Create auth module
|/
* pqr1234 Initial commit
8

Melihat Perbedaan: git diff

git diff menampilkan perbedaan antara versi file. Berguna untuk review perubahan sebelum commit atau membandingkan branch.

Simbol dalam Diff:
- (merah) = baris dihapus
+ (hijau) = baris ditambahkan
@@ = lokasi perubahan (hunk header)
Terminal
# Diff: working vs staging
git diff

# Diff: staging vs repo (yang akan di-commit)
git diff --staged

# Diff file tertentu
git diff index.html

# Diff antara 2 commit
git diff abc123 def456

# Diff antar branch
git diff main..feature/auth

# Diff hanya nama file yang berubah
git diff --name-only

# Diff dengan statistik
git diff --stat
Contoh Output Diff
diff --git a/index.html b/index.html
index abc1234..def5678 100644
--- a/index.html
+++ b/index.html
@@ -5,7 +5,9 @@
 <body>
   <h1>Hello World</h1>
-  <p>Welcome</p>
+  <p>Welcome to my website</p>
+  <p>This is a new paragraph</p>
+  <button>Click me</button>
 </body>
9

Mengabaikan File: .gitignore

File .gitignore memberitahu Git file/folder mana yang harus diabaikan. File yang di-ignore tidak akan muncul di git status dan tidak akan di-commit.

File yang Harus Di-ignore:
1. node_modules/ — dependency Node.js (besar, bisa di-install ulang)
2. .env — environment variables (berisi secret!)
3. dist/, build/ — hasil build (bisa di-generate ulang)
4. .DS_Store — file sistem macOS
5. *.log — file log
6. IDE settings — .vscode/, .idea/
.gitignore
# Dependencies
node_modules/
vendor/

# Build output
dist/
build/
.next/

# Environment files (PENTING!)
.env
.env.local
.env.*.local

# Logs
*.log
npm-debug.log*

# OS files
.DS_Store
Thumbs.db

# IDE
.vscode/
.idea/
*.swp

# Compiled files
*.class
*.exe
*.o

# Archive
*.zip
*.tar.gz
Tip: Kunjungi gitignore.io untuk generate .gitignore sesuai tech stack project kamu.
10

Branch: Bekerja Paralel

Branch adalah cabang independen dari development. Main branch (biasanya 'main' atau 'master') adalah kode produksi. Branch lain digunakan untuk pengembangan fitur tanpa mengganggu kode utama.

Mengapa Branching?
1. Fitur baru tidak mengganggu main branch
2. Bisa development paralel dengan tim
3. Eksperimen tanpa takut rusak
4. Code review sebelum merge
5. Bisa switch antar fitur kapan saja
Terminal
# Lihat semua branch
git branch

# Buat branch baru
git branch feature/login

# Pindah ke branch
git checkout feature/login

# Buat + pindah (shortcut)
git checkout -b feature/login

# Alternatif: switch (Git 2.23+)
git switch feature/login
git switch -c feature/login   # create + switch

# Hapus branch
git branch -d feature/login   # safe delete
git branch -D feature/login   # force delete

# Rename branch
git branch -m old-name new-name

# Lihat branch + commit terakhir
git branch -v
       main        ●────●────●────● (production)
                    \         /
feature/login        ●──●──●──┘    (fitur login)
                         \
feature/cart             ●──●      (fitur keranjang)

HEAD → menunjukkan branch aktif
11

Merge: Menggabungkan Branch

git merge menggabungkan perubahan dari satu branch ke branch lain. Biasanya merge dari feature branch ke main branch setelah fitur selesai.

Strategi Merge:
1. Fast-forward — Tidak ada commit baru di main, Git hanya memindahkan pointer
2. Three-way merge — Ada commit baru di kedua branch, Git membuat commit merge baru
Terminal
# Pastikan di branch tujuan (main)
git checkout main

# Merge feature branch ke main
git merge feature/login

# Merge dengan pesan custom
git merge feature/login -m "Merge login feature"

# Merge dengan no-ff (selalu buat merge commit)
git merge --no-ff feature/login

# Abort merge jika ada konflik
git merge --abort

# Setelah merge, hapus branch yang sudah tidak dipakai
git branch -d feature/login
BEFORE MERGE:
main        ●──●
                 \
feature/login     ●──●──●

AFTER MERGE (fast-forward):
main        ●──●──●──●──●

AFTER MERGE (--no-ff):
main        ●──●───────●  (merge commit)
                 \   /
feature/login     ●──●
12

Menyelesaikan Merge Conflicts

Conflict terjadi ketika dua branch mengubah baris yang sama di file yang sama. Git tidak bisa memutuskan mana yang benar, jadi meminta kamu untuk menyelesaikannya manual.

Langkah Selesaikan Conflict:
1. Git berhenti di tengah merge
2. Buka file yang konflik
3. Cari bagian dengan <<<<<<<
4. Edit sesuai keinginan (pilih salah satu atau gabungkan)
5. Hapus marker conflict
6. git add file yang sudah diedit
7. git commit (atau git merge --continue)
File dengan Conflict
<<<<<<< HEAD
const theme = 'dark';
=======
const theme = 'light';
>>>>>>> feature/theme

<!-- PENJELASAN: -->
<<<<<<< HEAD    = versi di branch aktif (main)
=======           = pemisah
>>>>>>> branch   = versi dari branch yang di-merge

<!-- SOLUSI: edit menjadi salah satu atau gabungan -->
const theme = 'dark';  <-- pilih salah satu
Terminal - Setelah Edit
# Setelah mengedit file, add dan commit
git add script.js
git commit -m "Resolve theme conflict"

# Atau gunakan merge --continue
git merge --continue

# Jika ingin batal merge
git merge --abort

# Lihat file yang konflik
git status
Tip: Gunakan VS Code atau editor modern yang memiliki fitur visual merge conflict resolution. Klik "Accept Current", "Accept Incoming", atau "Accept Both".
13

Remote Repository: GitHub

Remote adalah versi repository yang disimpan di server (seperti GitHub). Memungkinkan kolaborasi, backup, dan akses dari mana saja.

Konsep Local vs Remote:
Local → repository di komputermu
Remote → repository di server (GitHub, GitLab, dll)

Sinkronisasi: push (local → remote), pull (remote → local)
Terminal
# Lihat remote yang terdaftar
git remote -v

# Tambah remote (biasanya bernama 'origin')
git remote add origin https://github.com/user/repo.git

# Ubah URL remote
git remote set-url origin https://github.com/user/new-repo.git

# Hapus remote
git remote remove origin

# Lihat detail remote
git remote show origin

# Clone repository dari GitHub
git clone https://github.com/user/repo.git
git clone git@github.com:user/repo.git  # SSH
     git push          git pull
        │                 ↑
        ↓                 │
┌─────────────┐    ┌─────────────┐
│   LOCAL     │ ←→ │   REMOTE    │
│  (Komputer) │    │  (GitHub)   │
└─────────────┘    └─────────────┘
        ↑                 │
        │    git clone    │
        └─────────────────┘
14

Push & Pull: Sinkronisasi

git push mengirim commit local ke remote. git pull mengambil commit dari remote ke local. Ini adalah cara utama berbagi kode dengan tim.

Terminal
# Push ke remote (pertama kali, set upstream)
git push -u origin main

# Push selanjutnya (sudah ada upstream)
git push

# Push semua branch
git push --all origin

# Force push (HATI-HATI!)
git push --force

# Pull dari remote
git pull origin main

# Pull dengan rebase (lebih bersih)
git pull --rebase origin main

# Fetch tanpa merge (cek dulu)
git fetch origin
git log origin/main  # lihat apa yang akan di-pull
fetch vs pull:
git fetch hanya mengambil data tanpa merge — aman untuk cek perubahan
git pull = fetch + merge — langsung menggabungkan perubahan
15

Pull Request & Code Review

Pull Request (PR) adalah mekanisme di GitHub untuk mengusulkan perubahan. Tim bisa review kode, diskusi, dan approve sebelum di-merge ke branch utama.

Workflow Pull Request:
1. Buat branch fitur dari main
2. Commit perubahan di branch fitur
3. Push branch ke GitHub
4. Buka Pull Request di GitHub
5. Reviewer cek kode, kasih feedback
6. Diskusi dan revisi jika perlu
7. Approve dan merge ke main
8. Hapus branch fitur
Terminal
# Alur lengkap Pull Request

# 1. Buat branch dari main yang terbaru
git checkout main
git pull origin main
git checkout -b feature/user-profile

# 2. Kerjakan fitur
git add .
git commit -m "Add user profile page"

# 3. Push branch ke GitHub
git push -u origin feature/user-profile

# 4. Buka GitHub, klik "Compare & pull request"
# 5. Isi judul dan deskripsi PR
# 6. Tambahkan reviewer
# 7. Setelah approved, klik "Merge pull request"

# 8. Hapus branch lokal setelah merge
git checkout main
git pull origin main
git branch -d feature/user-profile
16

Stash: Menyimpan Perubahan Sementara

git stash menyimpan perubahan yang belum di-commit secara sementara. Berguna saat harus switch branch tapi belum siap commit.

Kapan Pakai Stash?
1. Ingin switch branch tapi ada perubahan belum selesai
2. Ingin pull dari remote tapi ada conflict dengan local
3. Ingin coba sesuatu tanpa commit dulu
4. Ingin simpan "draft" perubahan
Terminal
# Simpan perubahan ke stash
git stash

# Stash dengan pesan
git stash save "Work in progress on login"

# Lihat semua stash
git stash list

# Lihat isi stash tertentu
git stash show -p stash@{0}

# Ambil stash terakhir (dan hapus dari list)
git stash pop

# Ambil stash tapi tetap simpan
git stash apply

# Ambil stash tertentu
git stash apply stash@{2}

# Hapus stash
git stash drop stash@{0}

# Hapus semua stash
git stash clear
17

Reset & Revert: Membatalkan Perubahan

Ada beberapa cara membatalkan perubahan di Git, tergantung situasi dan apakah sudah di-push atau belum.

Kapan Pakai Apa?
git checkout -- → Batalkan perubahan di working dir (belum staged)
git reset HEAD → Unstage file (kembali ke modified)
git reset --soft → Undo commit, tapi simpan changes di staging
git reset --mixed → Undo commit, simpan changes di working dir
git reset --hard → Undo commit, HAPUS SEMUA changes (bahaya!)
git revert → Buat commit baru yang membatalkan commit lama (aman untuk yang sudah push)
Terminal
# Batalkan perubahan file (belum staged)
git checkout -- file.txt
git restore file.txt  # Git 2.23+

# Unstage file
git reset HEAD file.txt
git restore --staged file.txt  # Git 2.23+

# Undo commit terakhir (simpan changes)
git reset --soft HEAD~1

# Undo commit terakhir (changes di working dir)
git reset --mixed HEAD~1

# Undo commit terakhir (HAPUS SEMUA)
git reset --hard HEAD~1

# Revert commit (untuk yang sudah di-push)
git revert abc123

# Revert beberapa commit
git revert HEAD~3..HEAD
Peringatan: git reset --hard akan menghapus semua perubahan yang belum di-commit. Tidak bisa dikembalikan! Pastikan paham sebelum menjalankan.
18

SSH Key untuk GitHub

SSH Key memungkinkan autentikasi tanpa memasukkan username/password setiap push. Lebih aman dan praktis.

Terminal
# 1. Generate SSH key
ssh-keygen -t ed25519 -C "email@example.com"

# 2. Start SSH agent
eval "$(ssh-agent -s)"

# 3. Add key ke agent
ssh-add ~/.ssh/id_ed25519

# 4. Copy public key
cat ~/.ssh/id_ed25519.pub
# atau
clip < ~/.ssh/id_ed25519.pub  # Windows
pbcopy < ~/.ssh/id_ed25519.pub  # macOS

# 5. Tambah ke GitHub: Settings → SSH Keys → New SSH key

# 6. Test koneksi
ssh -T git@github.com

# 7. Ubah remote URL ke SSH
git remote set-url origin git@github.com:user/repo.git

Git Workflow: Best Practices

Workflow adalah aturan bagaimana tim menggunakan Git. Beberapa workflow populer:

1. Feature Branch Workflow (paling umum)
- Setiap fitur di branch terpisah
- Pull request untuk merge
- Code review wajib

2. Gitflow (release cycle formal)
- main: production code
- develop: integration branch
- feature/*: fitur baru
- release/*: persiapan release
- hotfix/*: bugfix urgent

3. Trunk-Based Development (CI/CD)
- Semua commit ke main
- Feature flags untuk fitur belum selesai
- Butuh CI/CD yang kuat 
GITFLOW:

main        ●────●────●────● (production)
            ↑    ↑    ↑
release/*   ─────●────┘
            ↑
develop  ●──●──●──●──●──●──●
         ↑       ↑
feature/ ●──●──● └──●──●──● (fitur A & B)
Best Practices:
- Commit sering, commit kecil
- Pesan commit jelas dan deskriptif
- Pull sebelum push
- Review code sebelum merge
- Hapus branch yang sudah di-merge
- Jangan commit file sensitif (.env, credentials)

Praktik: Setup Project dengan Git

Praktikkan semua yang sudah dipelajari dengan membuat repository lengkap dari awal hingga push ke GitHub.

Terminal - Full Workflow
# 1. Buat project
mkdir my-portfolio
cd my-portfolio

# 2. Init Git
git init

# 3. Buat file
touch index.html style.css README.md .gitignore

# 4. Isi .gitignore
echo ".DS_Store\n*.log" > .gitignore

# 5. Stage dan commit
git add .
git commit -m "Initial commit"

# 6. Buat di GitHub, lalu hubungkan
git remote add origin git@github.com:user/my-portfolio.git
git push -u origin main

# 7. Buat branch untuk fitur
git checkout -b feature/header

# 8. Edit, commit, push branch
git add index.html
git commit -m "Add header section"
git push -u origin feature/header

# 9. Buka Pull Request di GitHub
# 10. Merge, lalu update lokal
git checkout main
git pull origin main
